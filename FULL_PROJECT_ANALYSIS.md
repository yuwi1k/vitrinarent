# Полный анализ проекта Vitrina Rent

**Дата:** март 2026  
**Фокус:** сайт для агентов (контактов владельца платформы на публичных страницах быть не должно).

Анализ выполнен с трёх сторон: **разработчик**, **клиент** (агент или конечный пользователь каталога), **менеджер** (оператор дашборда). В конце — сводные таблицы и приоритеты.

---

# Часть 1: Взгляд разработчика

## 1.1 Архитектура и стек

| Компонент | Технология |
|-----------|------------|
| Backend | FastAPI |
| БД | PostgreSQL (asyncpg + psycopg2 для Alembic/SQLAdmin) |
| ORM | SQLAlchemy 2.x |
| Шаблоны | Jinja2 |
| Админка | SQLAdmin + кастомный дашборд |
| Фронт | HTML/CSS/JS, Bootstrap-подобная вёрстка, Tabler в дашборде |

**Плюсы:** чёткое разделение публичный сайт / дашборд / SQLAdmin, асинхронные роуты, единая работа с файлами через `file_utils`, миграции в одном месте, health-check для деплоя.

**Что починить / улучшить:**

1. **Два источника фида Авито**  
   - Публичный `/avito.xml` использует упрощённый `app/feed.py`.  
   - Выгрузка из дашборда `/dashboard/export/avito` использует полную логику в `dashboard.py` (_build_avito_feed_xml).  
   - Разные наборы полей и шаблоны → риск рассинхронизации.  
   **Рекомендация:** один модуль генерации фида (например, в `app/feed.py` или отдельном сервисе), а публичный роут и экспорт дашборда вызывают его с разными опциями (полный/упрощённый). Документировать разницу, если два формата намеренно.

2. **Поиск и иерархия объектов**  
   В `build_search_query` нет фильтра `parent_id.is_(None)`. В выдаче могут попадать и здания, и дочерние помещения. Для каталога обычно показывают либо только корневые объекты, либо явно различают здание/помещение.  
   **Рекомендация:** в публичном поиске фильтровать по `parent_id.is_(None)` (только корневые), либо добавить в UI тип (здание/помещение) и фильтр.

3. **Бизнес-логика в роутерах**  
   Создание/обновление объекта, загрузка файлов, экспорт Авито сосредоточены в `dashboard.py` и `routers.py`. При росте кода сложнее тестировать и переиспользовать.  
   **Рекомендация:** вынести в слой сервисов (например `property_service`, `avito_export_service`).

4. **Зависимости**  
   В `requirements.txt` может быть aiosqlite при использовании только PostgreSQL — удалить неиспользуемые зависимости.

## 1.2 Безопасность

| Что сделано | Что не сделано / риск |
|-------------|------------------------|
| Production: проверка SESSION_SECRET_KEY и ADMIN_PASSWORD | Rate limiting на логин и тяжёлые endpoint’ы |
| Санитизация HTML (bleach) в описаниях | Загрузки: нет проверки UPLOAD_MAX_FILE_SIZE и белого списка MIME/расширений |
| Ключи и контакты из .env | Нет CSRF-токенов для форм дашборда |
| Доступ в дашборд по сессии is_admin | Нет срока жизни сессии и инвалидации при смене пароля |

**Рекомендации:**

- Ввести rate limiting (например slowapi) на `/admin/login` и при необходимости на поиск и экспорт.
- В обработчиках загрузки: проверять размер тела запроса и разрешённые расширения/MIME для фото и документов.
- Рассмотреть CSRF для форм дашборда и ограничение TTL сессии.

## 1.3 Контакты и позиционирование «для агентов»

**Важно:** сайт позиционируется как сервис для агентов; контактов владельца платформы (ваших) на публичных страницах быть не должно.

- **Публичные страницы (главная, поиск, карточка объекта, FAQ):** телефона, email, формы «Оставить заявку» нет — это **соответствует** требованию.
- **Фид Авито:** контакт (телефон/имя менеджера) в XML нужен для объявления на Авито и задаётся через .env (AVITO_CONTACT_PHONE, AVITO_MANAGER_NAME). Это контакт агентства для объявлений, а не «наш» контакт на сайте — оставить как есть, настраивается через .env.

**Что убрать / не добавлять:**

- Не добавлять в футер или на карточку объекта телефон/email владельца платформы и кнопки «Позвонить нам» / «Оставить заявку».
- Если позже появятся контакты (например для поддержки агентов), делать их явно в отдельном закрытом разделе или только в дашборде.

**Что можно добавить (текст, без контактов):**

- На карточке объекта или в футере нейтральная подсказка: «По вопросам по объекту обращайтесь к вашему агенту» — без телефона и ссылок на вас.

# Часть 2: Взгляд клиента (агент / посетитель каталога)

Под «клиентом» здесь понимается тот, кто смотрит публичный каталог: агент, подбирающий объекты для своих клиентов, или конечный арендатор/покупатель, которому агент прислал ссылку.

## 2.1 Контент и позиционирование

- Заголовок и описание в духе «сервис для агентов» и каталога коммерческой недвижимости — уместны.
- Герой на главной: «Найдите идеальное пространство для вашего бизнеса», «Проверенные офисы, склады и торговые площади от собственника» — звучит скорее как B2C. Для каталога для агентов можно сместить акцент: например «Каталог объектов для ваших клиентов» или «Подборки коммерческой недвижимости для агентов».

**Рекомендация:** слегка скорректировать тексты главной под аудиторию «агент» (подборки, работа с объектами, передача ссылок клиентам), не добавляя ваших контактов.

## 2.2 Что убрать

- Любые намёки на «позвоните нам» / «напишите нам» с вашими контактами — таких нет, сохранять как есть.
- Разночтение в футере (см. п. 1.4) — убрать «Vitrina Real Estate System. Внутренний каталог» с публичной карточки и унифицировать.

## 2.3 Что добавить (без контактов)

- На карточке объекта короткий текст: «По вопросам по объекту обращайтесь к вашему агенту» (или аналогичный), без телефона/email.
- В FAQ уже есть блок «Для агентов»; при наличии блока «Для арендаторов» можно добавить одну строку: что по объекту обращаются к агенту, который прислал ссылку.

## 2.4 UX и доступность

- Есть skip-link, aria, focus-visible, lazy loading изображений — хорошо.
- Поиск: фильтры, сортировка, пагинация, карта — логика ясная.
- Карточка объекта: галерея, описание, документы, навигация по помещениям в здании — структура понятная.

**Рекомендации:** сохранять текущий уровень доступности; при доработках не убирать skip-link и семантику.

# Часть 3: Взгляд менеджера (оператор дашборда)

## 3.1 Что уже хорошо

- Стартовая страница со счётчиками (всего/активные/на главной, аренда/продажа).
- Список объектов: фильтры, поиск, пагинация, массовые действия (активация, «на главную»).
- Создание/редактирование/копирование объекта, вкладка Авито, загрузка фото и документов, порядок фото (drag-and-drop).
- Файловый менеджер по папкам объектов, загрузка в папку.
- Смена пароля из дашборда.
- Экспорт полного фида Авито (Продам/Сдам) с контактами из .env.

## 3.2 Что починить

- **Массовые действия и дочерние объекты**  
  Сейчас bulk-действия применяются только к объектам с `parent_id.is_(None)`. Нужно явно задокументировать или показывать в UI, что в списке только корневые объекты, а дочерние редактируются по отдельности (или добавить опцию «применить к дочерним» при необходимости).

- **Валидация загрузок**  
  Не проверяются размер файла (UPLOAD_MAX_FILE_SIZE) и тип (MIME/расширения). Стоит добавить проверку в endpoint’ах загрузки.

## 3.3 Что добавить

- **Настройки в UI**  
  Сейчас контакты для фида и при необходимости для сайта задаются только в .env. Для менеджера удобнее раздел «Настройки» в дашборде: имя менеджера, телефон для Авито (и при необходимости отображение на сайте, если позже решите показывать контакт только в личном кабинете). Хранить в БД или конфиг-файле, без обязательной правки .env.


- **Краткая справка по фиду**  
  В дашборде рядом с кнопкой «Скачать фид Авито» — 1–2 предложения: что в фиде (контакты из настроек, только активные корневые объекты и т.д.).

# Часть 5: Сравнение с типичными решениями

- **Каталог для агентов (B2B):** обычно нет общих контактов «компании» на публичных страницах; контакт — у агента или в объявлении на площадке (Авито). Ваша текущая модель (без контактов на сайте, контакт в фиде из .env) этому соответствует.
- **Порталы недвижимости (ЦИАН, Авито):** на карточке объявления контакт продавца/агента. У вас контакт на сайте не показывается — корректно для «каталог для агентов», чтобы лиды шли агенту, а не владельцу платформы.
- **Внутренние каталоги (CRM, экстранеты):** часто есть дашборд, экспорт, управление объектами. У вас дашборд и экспорт в порядке; усилить стоит настройки в UI и экспорт в CSV/Excel.

# Приоритеты (кратко)

1. **Обязательно:** не показывать ваши контакты на публичных страницах; унифицировать футер и один источник правды для фида Авито (или документировать два формата).
2. **Важно:** фильтр поиска по корневым объектам; проверка загрузок (размер, тип); rate limiting на логин.
3. **Желательно:** настройки в дашборде (контакты без .env), экспорт CSV/Excel, подсказка по фиду; .env.example, тесты, при необходимости Dockerfile и CI.
4. **По желанию:** слой сервисов, CSRF и TTL сессии, более явные тексты «для агентов» на главной.

---

**Итог:** проект логично ориентирован на агентов; отсутствие ваших контактов на публичном сайте соответствует задаче. Основные зоны доработки: единый бренд и фид, безопасность (загрузки, rate limiting), удобство менеджера (настройки, экспорт) и поддерживаемость кода (поиск, тесты, опционально сервисный слой).
