# Полный анализ проекта Vitrina Real Estate (обновлённый)

**Дата:** февраль 2026  
**Масштаб:** каталог коммерческой недвижимости (B2B), дашборд для менеджеров, фиды Авито (Продам / Сдам).

---

## Краткое резюме

Проект — каталог объектов с публичным поиском, картой (Яндекс.Карты), карточками с ЧПУ (slug), кастомным дашбордом (Tabler), файловым менеджером и выгрузкой фида Авито по официальным шаблонам. Стек: FastAPI, SQLAlchemy async, PostgreSQL, Jinja2, Bootstrap/Tabler. После предыдущего анализа внедрены: health-check, тесты, санитизация HTML, ключ карт из .env, сортировка поиска, meta/OG, доступность, стартовая страница дашборда, фильтр и массовые действия в списке, копирование объекта, смена пароля из дашборда, превью фото, перетаскивание порядка фото, полный фид Авито (отдельно Продам/Сдам) и вкладка «Авито» в форме. Для продакшна по-прежнему желательны: README, Dockerfile, rate limiting, контакты/CTA на сайте, опционально — экспорт в CSV/Excel.

# Часть 1: Взгляд разработчика

## 1.1 Архитектура и код

**Сделано:**
- Разделение: `routers.py` (публичные маршруты), `dashboard.py` (дашборд + фид Авито), `admin_views.py` (SQLAdmin), `services.py` (поиск), `file_utils.py` (пути, ресайз, нормализация URL).
- Асинхронные сессии (AsyncSession) в роутерах и дашборде.
- Миграции Alembic в `migrations/` (PostgreSQL).
- Модели: Property (slug, avito_object_type, avito_data JSON), PropertyImage (sort_order), PropertyDocument.
- Фид Авито в дашборде: два шаблона (Продам / Сдам), порядок тегов по официальным шаблонам, CDATA для Description/VideoFileURL/EgrnExtractionLink.

**Минусы и риски:**
- Дублирование: в проекте есть и `migrations/`, и `alembic/` (alembic/versions, alembic/env.py). Используется `migrations/`; папку `alembic/` при желании можно удалить или не трогать.
- Публичный фид `app/feed.py` — упрощённый (без полного набора тегов Авито). Основная выгрузка идёт из дашборда (`/dashboard/export/avito`). Имеет смысл либо развивать один источник фида, либо явно документировать, что публичный `/avito.xml` — базовая версия.
- Константы: `PAGE_SIZE` в `routers.py` (12) и в `dashboard.py` (20) — можно вынести в конфиг.
- Нет слоя сервисов для сложной бизнес-логики: создание/обновление объекта и загрузка файлов остаются в роутерах дашборда. При росте функционала стоит вынести в отдельные сервисы.

**Рекомендации:**
- Оставить одну точку миграций (`migrations/`), при необходимости почистить `alembic/`.
- Константы пагинации и лимиты загрузок — в .env или Pydantic Settings.
- При расширении — слой сервисов (property_service, avito_export_service и т.д.).

## 1.2 Безопасность

**Сделано:**
- Ключ Яндекс.Карт в `.env` (`YANDEX_MAPS_API_KEY`), передаётся в шаблон поиска через контекст; в шаблоне подставляется только при наличии ключа.
- Описание объекта: вывод через фильтр `sanitize_html` (bleach), белый список тегов (p, br, strong, em, ul, ol, li, a и т.д.) — защита от XSS.
- Проверка production-секретов: при `ENVIRONMENT=production` запрещены дефолтные `SESSION_SECRET_KEY` и `ADMIN_PASSWORD`; пароль админки можно менять из дашборда (файл `data/.admin_password` имеет приоритет над .env).
- Контакты для фида Авито и при необходимости для сайта — из .env (`AVITO_CONTACT_PHONE`, `AVITO_MANAGER_NAME`).

**Остаётся:**
- Rate limiting на `/admin/login` и при необходимости на тяжёлые эндпоинты (поиск, экспорт фида) — не реализован. Рекомендуется slowapi или middleware.
- Загрузка файлов: ограничение размера тела запроса и белый список MIME/расширений на бэкенде — желательно добавить.
- CORS в `main.py` не настроен. При появлении SPA или запросов с другого домена — задать явный `allow_origins`, не использовать `"*"` в продакшне.
- Сессия админки: только проверка `request.session.get("is_admin")`. При необходимости — срок жизни сессии, привязка к IP или токену.

**Рекомендации:**
- Добавить rate limiting на логин и опционально на экспорт фида.
- Ограничить размер и типы загружаемых файлов на бэкенде.

## 1.3 Тестирование и CI/CD

**Сделано:**
- Папка `tests/`: `conftest.py` (TestClient), `test_routes.py` — тесты на `/health/liveness`, `/health/readiness`, главную, поиск, карточку объекта (404), FAQ, редирект дашборда без сессии.
- В `requirements.txt`: pytest, pytest-asyncio, httpx.
- Тесты используют синхронный TestClient; для полноценной проверки async-роутов с БД при необходимости можно перейти на httpx.AsyncClient и фикстуры с тестовой БД.

**Остаётся:**
- Нет CI (GitHub Actions / GitLab CI).
- Нет линтеров/форматтеров в пайплайне (ruff, black, mypy).
- Нет тестов на `build_search_query`, генерацию фида Авито, дашборд (создание/редактирование объекта).

**Рекомендации:**
- В CI: установка зависимостей, линтер (ruff), прогон тестов.
- Опционально: тесты на `build_search_query` и экспорт фида (с моками или тестовой БД).

## 1.4 DevOps и деплой

**Сделано:**

- `docker-compose.yml` для PostgreSQL.
- Переменные окружения описаны в `.env.example`.

**Остаётся:**
- Нет Dockerfile для приложения, нет образа с uvicorn/gunicorn.
- В docker-compose только сервис `db`; сервиса `app` с миграциями и запуском приложения нет.
- Нет README с описанием проекта, установки, переменных окружения, запуска миграций и сервера.
- Нет описанной схемы деплоя (systemd, Docker, Nginx).

**Рекомендации:**
- Добавить README: описание, установка, .env, миграции (`alembic upgrade head`), запуск (`uvicorn app.main:app`).
- Dockerfile (multi-stage) и сервис `app` в docker-compose с зависимостью от `db` и командой миграций при старте.
- В продакшне: Nginx перед приложением (TLS, статика, лимиты).

## 1.5 Зависимости и техдолг

- `requirements.txt`: версии частично зафиксированы; для воспроизводимости сборки лучше зафиксировать все (pip freeze или poetry/uv).

- Два интерфейса: SQLAdmin (минимальный) и кастомный Dashboard. Логично развивать только Dashboard; SQLAdmin оставлен для отладки/резерва.

**Что добавить (по необходимости):**

# Часть 3: Взгляд пользователя — менеджер (дашборд)

## 3.1 Текущие возможности

**Сделано:**
- Вход через SQLAdmin (`/admin/login`), сессия общая с дашбордом; смена пароля из дашборда (`/dashboard/settings/password`), приоритет файла `data/.admin_password`.
- Стартовая страница дашборда (`/dashboard`): счётчики (всего, активных, на главной, аренда/продажа), быстрые действия (Объекты, Файлы, Создать объект).
- Объекты: список с пагинацией, фильтр (поиск по названию/адресу, тип сделки, категория, статус), массовые действия (опубликовать, снять с публикации, на главную, убрать с главной), кнопка «Копировать» для копирования объекта в форму создания.
- Создание/редактирование: вкладки Основное, Коммерция, Медиа, Авито, Настройки. Карта для координат (Яндекс.Карты). Превью выбранных фото перед загрузкой. Перетаскивание порядка фото в галерее (Sortable.js), сохранение порядка через AJAX.
- Файловый менеджер: список папок (объектов), вход в папку, загрузка фото/документов, удаление по AJAX, превью выбранных фото.
- Экспорт фида Авито: кнопка «Скачать фид Авито» в шапке дашборда (`/dashboard/export/avito`). Два шаблона по типу сделки: «Продам» и «Сдам» — только релевантные теги, порядок по официальным шаблонам Авито. В форме объекта вкладка «Авито»: обязательные и необязательные поля, разделение полей по типу сделки (при смене «Тип сделки» показываются только поля для Продам или для Сдам).

---

## 3.2 Чего не хватает менеджеру

**Отчёты и экспорт:**
- Нет экспорта списка объектов в Excel/CSV для отчётов.
- Нет аналитики (просмотры карточки, источники трафика) — по желанию.

**Настройки:**
- Телефон и имя менеджера для фида и для отображения на сайте — только из .env. Нет раздела «Настройки сайта» в дашборде (контакты, логотип и т.д.) с сохранением в БД или файле.

**Файловый менеджер:**
- Нет переименования документа (только имя при загрузке).
- Нет превью PDF в списке файлов (опционально).
- Загрузка в папке объекта через форму с редиректом после отправки; можно оставить так или добавить AJAX-загрузку без перезагрузки.

**Рекомендации:**
- Опционально: экспорт списка объектов в CSV/Excel.
- Опционально: раздел «Настройки» в дашборде (контакты для сайта и фида, если не хотите править .env вручную).
